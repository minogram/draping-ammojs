결론부터 말씀드리면, Jolt Physics가 현재 요구사항에 가장 부합하는 강력한 대안이며, 성능이 최우선이라면 WebGPU 기반 방식을 고려해 볼 수 있습니다.

1. Jolt Physics (Wasm) - 가장 추천하는 대안
최근 많은 웹 게임 엔진들이 Ammo.js에서 갈아타고 있는 엔진입니다.

해결책: XPBD(Extended Position Based Dynamics) 알고리즘을 사용합니다. 이 방식은 Ammo.js의 방식보다 수치적으로 훨씬 안정적이며, 시뮬레이션 도중 강성(Stiffness)이나 질량을 변경해도 리셋 없이 즉각 반영하기에 매우 유리합니다.
장점: 멀티코어 최적화가 잘 되어 있어 CPU 기반 엔진 중 성능이 가장 뛰어나며, 메모리 효율이 좋아 고해상도에서도 안정적입니다.
단점: 소프트 바디 기능이 비교적 최신이라 Three.js와의 통합 예제가 Ammo.js만큼 많지는 않습니다.
2. Rapier.js
Rust로 작성되어 성능과 안정성이 검증된 현대적인 엔진입니다.

해결책: FEM(유한요소법) 기반의 소프트 바디를 지원합니다. 이는 물리적으로 매우 정확한 변형을 보여줍니다.
장점: TypeScript 친화적인 API를 제공하여 내부 노드나 데이터에 접근하기가 매우 쉽습니다. Ammo.js처럼 get_m_links가 없어 고생할 일이 없습니다.
단점: FEM 방식은 연산량이 많아 150x150 이상의 초고해상도에서는 PBD 방식보다 느려질 수 있습니다.
3. WebGPU 기반 PBD (Custom/GPU Simulation)
라이브러리 형태보다는 특정 기술 스택에 가깝지만, 현재 겪고 계신 "해상도와 실루엣" 문제를 근본적으로 해결할 수 있는 유일한 방법입니다.

해결책: CPU가 아닌 GPU(Compute Shader)에서 수만 개의 정점을 동시에 계산합니다.
장점: 200x200 해상도에서도 60fps를 유지할 수 있습니다. VRAM을 사용하므로 시스템 메모리 부족(OOM) 문제에서 자유롭습니다.
단점: 물리 엔진이라기보다 "천 시뮬레이션 쉐이더"에 가까워, 장애물과의 충돌 처리를 직접 구현해야 하는 경우가 많습니다.
4. PhysX 5 (Wasm)
NVIDIA의 기술력이 집약된 엔진입니다.

장점: 기능적으로는 가장 완벽하며, 천 시뮬레이션 전용 최적화 옵션이 매우 많습니다.
단점: API가 매우 방대하고 무거우며, Wasm 버전은 GPU 가속을 사용할 수 없어 순수 CPU 성능만으로는 Jolt보다 느릴 수 있습니다.

다음 단계 제안:
만약 현재 프로젝트의 구조를 유지하면서 엔진만 교체하고 싶으시다면 Jolt Physics를 검토해 보시는 것을 가장 추천드립니다. 만약 상용 수준의 초고해상도 퀄리티가 목표라면 WebGPU를 이용한 커스텀 시뮬레이터 제작이 장기적으로 유리합니다.
